## Terminology Glossary

This section defines key terms used throughout the document. Terms are used consistently in architecture sections below.

**Table 1 — Terminology Glossary**  
(Columns: Term | Definition. Use consistent terminology from this table throughout the document.)

| Term | Definition |
|------|-------------|
| **Agent** | Autonomous software entity responsible for performing development tasks (see "[Role] Agent" naming below). |
| **Orchestration System / Orchestrator** | Central system coordinating workflows and agent execution. The Orchestrator is the runtime component of the Orchestration System (canonical: use "Orchestration System" for the layer, "Orchestrator" for the runtime). |
| **Task** | Unit of work executed by agents (lifecycle: see Canonical Task Lifecycle in Data Models). |
| **Workflow** | Structured sequence of tasks executed to achieve a development objective. |
| **Memory and Knowledge Layer** | Canonical term for the persistent knowledge system storing architectural insights, research, and codebase knowledge. |
| **Memory Agent** | Agent role that maintains and writes to the Memory and Knowledge Layer; stores institutional knowledge (see "[Role] Agent" naming). |
| **Codebase Understanding Agent** | Canonical name for the agent that maintains system-wide code knowledge; "Codebase Agent" is acceptable in context. |
| **Model Router** | System responsible for selecting AI models for inference. Agents obtain inference via Orchestrator → Model Router → Providers (agents never call models directly). |
| **Knowledge Graph / Dependency Graph** | Knowledge Graph: semantic/architectural relationships. Dependency Graph: code/module/service dependencies. Both are part of the Memory and Knowledge Layer and Codebase Understanding System. |
| **Timestamp convention** | Use created_at, updated_at (and optionally completed_at, assigned_at) consistently for event times. Prefer timestamp type; use created_at for record creation time. |
| **Agent role naming** | Use "[Role] Agent" (e.g., Backend Engineer Agent, Frontend Engineer Agent, QA Agent, Security Agent) when referring to the agent role. |
| **Sandbox Execution Environment** | Isolated execution environment for running generated code (canonical term; "Sandbox" acceptable in context). |
| **Artifact** | Build output generated by CI/CD pipelines. |
| **Namespace** | Logical isolation boundary used for project separation. Implemented by project_id (and optionally tenant_id); one tenant may have multiple projects. |
| **Development Infrastructure Layer** | Canonical term for CI/CD, build systems, and sandboxed execution (same as "Infrastructure Layer" in some diagrams). For production runtime, use "Deployment and Runtime Layer." |
| **Deployment Infrastructure** | Section/subsystem name (Section 14) for the subsystem that implements build, CI/CD, and deployment; it aligns with Development Infrastructure Layer (build/test) and Deployment and Runtime Layer (production). |
| **Knowledge Layer (in diagrams)** | In architecture diagrams, "Knowledge Layer" or "Shared Knowledge Infrastructure" denotes the Memory and Knowledge Layer (canonical term). |
| **Governance and Safety Layer** | Same as "Safety and Guardrail System" when referring to the layer; "Governance and Safety" is used in diagrams and Section 4; "Safety and Guardrail System" is the full subsystem name (Section 10). |
| **Idea Engine** | Conceptual source of requirement or opportunity detection; in the main architecture this function is performed by Monitoring agents, Product Manager Agent, and Research agents (see Section 13 — Autonomous Development Workflow, Stage 1 — Requirement or Opportunity Detection). |
| **Economic / Value Optimization System (EVOS)** | Subsystem (Section 22) that evaluates task value and prioritizes work; may be implemented by or integrated with an Economic Agent. "Economic/Value Agent" in Appendix A refers to this system or its implementing agent. |
